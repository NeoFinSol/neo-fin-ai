# Пользовательский контур НеоФин.Документы

## 1. Назначение пользовательского контура

Этот каталог — веб-клиент продукта **НеоФин.Документы**: он сопровождает полный **жизненный цикл анализа** от загрузки отчётности до просмотра сохранённого результата, а не служит только экраном «отправить файл».

Клиент:

- запускает анализ и показывает ход выполнения;
- получает итог в виде, пригодном для проверки (метрики, коэффициенты, скоринг, методология, метаданные извлечения);
- даёт повторный доступ к завершённым анализам через историю;
- отображает состояние **языкового слоя** по данным сервера, без подмены факта догадками по полю текстового блока.

Общее продуктовое позиционирование и состав системы — в корневом [`README.md`](../README.md); прикладной контракт сервера — в [`docs/API.md`](../docs/API.md); конфигурация окружений — в [`docs/CONFIGURATION.md`](../docs/CONFIGURATION.md); место клиента в архитектуре — в [`docs/ARCHITECTURE.md`](../docs/ARCHITECTURE.md); **пошаговый запуск на Windows** (Docker и нативный контур) — в [`docs/INSTALL_WINDOWS.md`](../docs/INSTALL_WINDOWS.md).

## 2. Роль фронтенда в общей архитектуре

Пользовательский контур в архитектуре системы отвечает за:

- сценарий работы человека с анализом (запуск, ожидание, просмотр, возврат к сохранённым задачам);
- **наблюдаемость** долгой обработки: параллельно используются канал WebSocket и периодический опрос HTTP (см. раздел 4);
- **представление объяснимости**: скоринг, методология, достоверность извлечения, корректные пояснения при недоступности или сбое языкового поставщика.

Числовой расчёт и извлечение выполняются на сервере; клиент не дублирует бизнес-логику, а отображает ответы API и состояния в согласовании с типами в `src/api/interfaces.ts`.

## 3. Основные пользовательские сценарии

Маршрутизация (`src/App.tsx`):

| Путь | Назначение |
|------|------------|
| `/login` | Ввод ключа доступа (`Auth`), сохранение в `localStorage` для заголовка `X-API-Key` |
| `/` | Рабочая область: загрузка PDF, выбор поставщика для языкового контура, прогресс, отчёт (`Dashboard`) |
| `/history` | Постраничный список анализов с сервера, открытие сохранённого отчёта (`AnalysisHistory`) |
| `/settings` | Параметры интерфейса, оформление профиля и справочные блоки (`SettingsPage`) |
| `/404` | Страница отсутствия маршрута |

**Главная (`Dashboard`):** один файл PDF за раз (`Dropzone`, до 50 МБ), выбор поставщика через `AiProviderMenu` (список доступных значений подгружается с `GET /system/ai/providers`; при наличии в списке `ollama` интерфейс по умолчанию предпочитает его, иначе — провайдер по умолчанию с сервера или `auto`). Пояснение на экране: переключатель влияет на языковой контур, **не** на детерминированный скоринг.

**Отчёт:** при успешном завершении на главной показывается `DetailedReport` с тем же составом данных, что и при открытии из истории.

**История:** запросы `GET /analyses`, детализация `GET /analyses/{task_id}`; при наличии тела отчёта — просмотр через `DetailedReport`.

**Многопериодная динамика в интерфейсе:** компонент `DetailedReport` и хук `useMultiAnalysisPolling` поддерживают опрос `GET /multi-analysis/{session_id}` и вкладку «Динамика» с графиком периодов при переданном `multiSessionId`. В текущих потоках с главной страницы и из истории **`multiSessionId` не передаётся**; на вкладке «Динамика» в этом случае показывается текст-подсказка о сравнении периодов. Полноценный сценарий с сессией должен быть согласован с серверным API и вызывающим кодом (см. [`docs/API.md`](../docs/API.md)).

## 4. Как интерфейс работает со статусами анализа

Логика однотипного анализа сосредоточена в `src/context/AnalysisContext.tsx`:

1. **Загрузка:** `POST /upload` (multipart), ответ с `task_id`.
2. **Параллельно:**  
   - **Опрос:** цикл `GET /result/{task_id}` с интервалом 2 с (верхняя граница попыток рассчитана на длительные задачи, в том числе с OCR).  
   - **WebSocket:** при статусе «обработка» подключается хук `useAnalysisSocket` к адресу `ws(s)://<хост>/api/ws/{task_id}` (в разработке префикс `/api` совпадает с прокси Vite к бэкенду, см. `vite.config.ts`).

Завершение по опросу или по сообщению WebSocket приводит к одному и тому же отображению результата. Требования к аутентификации HTTP и WebSocket на конкретной среде нужно сверять с [`docs/API.md`](../docs/API.md).

На главной во время ожидания показываются этапы (загрузка, извлечение, расчёт, языковой блок) в виде временной шкалы; это **организация интерфейса**, а не гарантированная пошаговая телеметрия сервера.

## 5. Как интерфейс отображает результат и объяснимость

`DetailedReport` объединяет:

- **Скоринг и методологию** — `ScoreInsightsCard`: итоговый балл, уровень риска, достоверность скоринга, значки профиля сравнения и базы периода, раскрываемый блок «Как рассчитано» (`score.methodology`: в том числе `leverage_basis`, `ifrs16_adjusted`, `adjustments`, `peer_context` при наличии в ответе).
- **Коэффициенты и диаграммы** — блок финансовых коэффициентов с `@mantine/charts` / `recharts`.
- **Исходные метрики и извлечение** — `DetailedMetricsCard` с `ConfidenceBadge` по `extraction_metadata` для ключевых полей; при низкой `confidence_score` отчёта — предупреждение по порогу из `constants/report.ts`.

Таким образом, интерфейс опирается не только на итоговый балл, а на **слой объяснимости**, который сервер уже включил в модель ответа.

## 6. Работа с языковым слоем и `ai_runtime`

Текстовые рекомендации и факторы приходят в поле `nlp` ответа, но **решение о показе «живого» языкового блока** принимается в `ScoreInsightsCard` так:

- показываются содержательные инсайты только если `ai_runtime.status === 'succeeded'` **и** в `nlp` есть непустые массивы рисков или ключевых факторов;
- для статусов `skipped`, `empty`, `failed` и при отсутствии `ai_runtime` выводятся **честные пояснения** и детерминированный контент (факторы скоринга, коэффициенты), без ложного сообщения «ИИ недоступен», если задача завершилась иначе.

Поле `ai_runtime` (`requested_provider`, `effective_provider`, `status`, `reason_code`) — **опорный источник** для текстов о состоянии языкового контура; поле `nlp` само по себе для этого не используется.

## 7. Структура фронтенд-проекта

| Путь | Содержание |
|------|------------|

## 9. Локальный запуск и разработка

Из корня репозитория:

```bash
npm --prefix frontend install
npm --prefix frontend run dev
```

Сервер разработки Vite слушает порт **3000** (`package.json`, скрипт `dev`). Запросы к API по префиксу `/api` проксируются на `http://localhost:8000` (см. `vite.config.ts`); WebSocket для статусов идёт через тот же префикс.

Сборка и проверка типов:

```bash
npm --prefix frontend run build
npm --prefix frontend run lint
```

Тесты (Vitest):

```bash
npm --prefix frontend run test
npm --prefix frontend run coverage
```

Пороги покрытия заданы в `vite.config.ts`; фактические цифры зависят от текущего состояния ветки — ориентируйтесь на вывод последнего прогона.

---

## 10. Источник правды для клиентских контрактов и типов

Каноническое описание структур ответа анализа, скоринга, `ai_runtime`, метаданных извлечения и вспомогательных сущностей — файл **`src/api/interfaces.ts`**.

При расхождении между комментарием в коде и ответом сервера при интеграции **приоритет у фактического JSON бэкенда**; типы на клиенте следует обновлять осознанно после изменения API (см. [`docs/API.md`](../docs/API.md)).

Файл `src/api/types.ts`, если присутствует, не позиционируется как основной контракт.

## 11. Примечания для сопровождения

- Ключ API для браузера хранится в **`localStorage`** под именем `neofin_api_key`; перехватчик Axios подставляет заголовок `X-API-Key`. Режим `DEV_MODE` на сервере ослабляет проверку — это вопрос конфигурации среды, а не клиента.
- Часть текстов на экране (название продукта, экосистема, юридическая строка в подвале) задаётся в **`src/constants/branding.ts`**; при смене публичной формулировки имеет смысл синхронизировать с корневым `README.md`.
- В `SettingsPage` разделы «Тариф» и часть блоков API-ключа могут носить **макетный** характер: перед опорой на них в продуктовой документации сверяйтесь с фактической политикой биллинга и выдачи ключей (в репозитории это может быть не отражено).
