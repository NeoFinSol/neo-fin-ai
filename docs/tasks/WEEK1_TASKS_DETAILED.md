## Неделя 1 — детализированные задачи и договорённости

Этот документ конкретизирует задачи недели 1 из общего плана и фиксирует договорённости по структуре репозитория и результатам недели.

### 1. Общие договорённости

- **Репозиторий**: `neo-fin-ai` (структура зафиксирована в `REPO_STRUCTURE.md` в корне).
- **Основные папки, которые должны существовать к концу недели 1**:
  - `backend/app/api/`, `backend/app/core/`, `backend/app/models/`
  - `frontend/`
  - `data/pdf_samples/`
  - `docs/`
- **Definition of Done для недели 1**:
  - Все перечисленные ниже задачи отмечены как выполненные (минимально, но работоспособно).
  - Можно: положить 1–2 тестовых PDF и пройти сквозной путь *«загрузка PDF на фронте → запрос в FastAPI → извлечённый текст в JSON → отображение на фронте»*.

---

### 2. Задачи по ролям (детализация)

#### 2.1. Капитан

- **K1. Утверждение структуры репозитория**
  - [ ] Ознакомиться с `REPO_STRUCTURE.md` и предложить коррективы (если нужны).
  - [ ] Принять финальную версию структуры вместе с командой.
  - [ ] Зафиксировать ссылку на репозиторий и структуру в командных материалах (чат, Notion и т.п.).

- **K2. Организация работы недели 1**
  - [ ] Провести короткий созвон/переписку с командой для walkthrough по задачам `WEEK1_TASKS_DETAILED.md`.
  - [ ] Уточнить, кто из программистов и аналитиков берёт какие подпункты и в какие сроки.
  - [ ] Договориться о формате ежедневного/еженедельного апдейта (например, короткий статус в чате в конце дня).

#### 2.2. Аналитик 1 (данные)

- **A1. Сбор тестовых PDF-отчётов**
  - [ ] Найти 5–7 отчётов (минимум: 2 скана и 2 текстовых PDF).
  - [ ] Сохранить их в `data/pdf_samples/` с понятными именами файлов (например, `company_year_type.pdf`).
  - [ ] В отдельной табличке (Google Sheets / Excel) указать:
    - компанию,
    - годы,
    - тип (`scan` / `text`),
    - краткое описание структуры.

- **A2. Список ключевых показателей**
  - [ ] Составить первичный список показателей (выручка, себестоимость, EBIT, EBITDA, чистая прибыль, активы, обязательства, капитал и т.д.).
  - [ ] Для каждого показателя:
    - описать возможные формулировки/синонимы в отчётах (для будущих regex),
    - указать тип значения (денежный / абсолютный / относительный).
  - [ ] Сохранить этот список в `docs/ANALYTICS_SPECS.md` в отдельном разделе «Ключевые показатели».

#### 2.3. Аналитик 2 (рынок, бизнес-модель)

- **A3. Сценарии использования ассистента**
  - [ ] Описать 3–4 сценария (например: быстрый чек финансов, анализ динамики, подготовка к совещанию).
  - [ ] Для каждого сценария указать:
    - кто пользователь,
    - что он вводит (PDF отчёт, название компании и т.п.),
    - какой результат хочет получить (цифры, выводы, рекомендации, PDF-отчёт и т.д.).
  - [ ] Добавить раздел «Сценарии использования» в `docs/ANALYTICS_SPECS.md` или отдельный документ по договорённости с капитаном.

- **A4. Первичные требования к скорингу и NLP**
  - [ ] Сформировать список групп коэффициентов (ликвидность, рентабельность, устойчивость, деловая активность и т.п.).
  - [ ] Предложить черновую схему интегрального скоринга (какие группы важнее, грубые веса).
  - [ ] Составить черновой перечень типов рисков и перспектив для анализа пояснительных записок (регуляторные, рыночные, операционные, кредитные и т.д.).
  - [ ] Зафиксировать это в `docs/ANALYTICS_SPECS.md` в разделах про скоринг и NLP.

#### 2.4. Программист 2 (бэкенд, ведущий)

- **B1. Инициализация структуры бэкенда**
  - [ ] Создать папки и файлы:
    - `backend/app/main.py`
    - `backend/app/api/endpoints.py`
    - `backend/app/core/ocr.py`
    - `backend/app/core/extractor.py`
    - `backend/app/core/ratios.py`
    - `backend/app/core/scoring.py`
    - `backend/app/core/recommendations.py`
    - `backend/app/core/nlp.py`
    - `backend/app/core/news.py`
    - `backend/app/models/schemas.py`
  - [ ] Во всех модулях ядра и `schemas.py` пока можно оставить заглушки (минимальные интерфейсы/функции без реализации).

- **B2. Базовый FastAPI-сервис**
  - [ ] В `main.py`:
    - создать приложение FastAPI,
    - добавить health-check эндпоинт `/health` (возвращает `"status": "ok"`).
  - [ ] В `api/endpoints.py`:
    - добавить заготовку эндпоинта `/analyze` (пока может возвращать тестовый JSON или просто принятый файл без анализа).
  - [ ] Подключить роуты из `api/endpoints.py` в `main.py`.

- **B3. Подготовка к OCR и извлечению текста**
  - [ ] В `core/ocr.py`:
    - описать интерфейс функции определения типа PDF (`is_scanned(pdf_path) -> bool`),
    - описать интерфейсы функций извлечения текста из сканов и текстовых PDF (без полной реализации).
  - [ ] В `core/extractor.py`:
    - описать интерфейс функции, которая принимает сырой текст и возвращает пока просто этот же текст или минимальную структуру (для проверки пайплайна).
  - [ ] В `schemas.py`:
    - определить черновую схему ответа для `/analyze`, включающую хотя бы поле `raw_text: str` и метаданные.

- **B4. Базовый `requirements.txt`**
  - [ ] Создать `backend/requirements.txt` с минимальным набором:
    - FastAPI, Uvicorn,
    - pdfplumber,
    - pytesseract,
    - pydantic,
    - requests (на будущее).

#### 2.5. Программист 1 (фронтенд)

- **F1. Стартовый фронтенд**
  - [ ] Создать файлы:
    - `frontend/index.html`
    - `frontend/style.css`
    - `frontend/script.js`
  - [ ] В `index.html`:
    - добавить форму загрузки одного PDF-файла,
    - добавить блок для отображения статуса/результата анализа.

- **F2. Интеграция с бэкендом**
  - [ ] В `script.js`:
    - реализовать отправку файла на эндпоинт `/analyze` через `fetch` + `FormData`,
    - отобразить полученный `raw_text` (или тестовый JSON) в текстовом блоке на странице.
  - [ ] В `style.css`:
    - добавить минимальную, но аккуратную стилизацию (читаемые шрифты, отступы).

### 3. Минимальный чек-лист готовности недели 1

- [ ] Структура репозитория зафиксирована в `REPO_STRUCTURE.md` и принята командой.
- [ ] Папки и файлы бэкенда созданы, FastAPI поднимается локально и отвечает на `/health`.
- [ ] Эндпоинт `/analyze` принимает PDF и возвращает хотя бы сырой текст или тестовый JSON.
- [ ] Базовый фронтенд умеет загрузить PDF и показать ответ бэкенда.
- [ ] В `data/pdf_samples/` лежат 5–7 реальных PDF-отчётов.
- [ ] В `docs/ANALYTICS_SPECS.md` зафиксированы: список ключевых показателей, первые сценарии использования, группы коэффициентов и типы рисков/перспектив.
