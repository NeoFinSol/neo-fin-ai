## Неделя 3 — NLP, рекомендации, новости и Docker

Этот документ конкретизирует задачи недели 3 из общего плана и фокусируется на:
- подключении локальной LLM (через Ollama) для анализа пояснительных записок;
- реализации модуля текстовых рекомендаций;
- (опционально) модуле новостей и анализа сентимента;
- подготовке Docker-конфигурации для более удобного запуска.

### 1. Цели недели 3 (уточнение)

- **NLP‑анализ**: уметь по тексту пояснительной записки выделять ключевые риски и перспективы и строить краткое резюме.
- **Рекомендации**: формировать структурированные рекомендации на основе числовых результатов (коэффициенты, скоринг) и NLP‑анализа.
- **Новости (опционально)**: при наличии времени — собирать краткую выборку новостей по компании и оценивать тональность.
- **Инфраструктура**: иметь рабочий `Dockerfile` для бэкенда и базовый `docker-compose.yml`, которые можно будет дополировать на неделе 4.

### 2. Задачи по ролям (детализация)

#### 2.1. Аналитик 1 (данные)

- **A3.1. Подготовка примеров для LLM**
  - [ ] Выбрать 3–5 примеров пояснительных записок (или фрагментов) из реальных или открытых источников.
  - [ ] Для каждого примера вручную:
    - выделить фрагменты, относящиеся к **рискам** (регуляторным, рыночным, операционным, кредитным и т.п.);
    - выделить фрагменты, относящиеся к **перспективам/возможностям** (рост рынка, новые продукты и т.д.);
    - указать, какие из них считаются **ключевыми** для краткого резюме.
  - [ ] Сохранить разметку в табличной форме (или JSON/Markdown) в `data/` (например, `data/nlp_examples_week3.*`), чтобы программист 2 мог использовать её для настройки промптов и тестов.

- **A3.2. Список типичных формулировок рисков и перспектив**
  - [ ] Совместно с аналитиком 2 выделить **набор типичных формулировок**:
    - для рисков (например, “существует риск”, “не исключается возможность”, “может негативно повлиять”);
    - для перспектив (например, “ожидается рост”, “расширение рынка”, “увеличение доли”).
  - [ ] Разбить формулировки по **категориям рисков/перспектив** (регуляторные, рыночные, операционные, кредитные и др.).
  - [ ] Зафиксировать это в `docs/ANALYTICS_SPECS.md` в разделе, посвящённом NLP и анализу пояснительных записок.

#### 2.2. Аналитик 2 (рынок, бизнес-модель)

- **A3.3. Логика рекомендаций на основе чисел и текстов**
  - [ ] На основе уже определённых коэффициентов и скоринга (неделя 2) сформировать **правила генерации рекомендаций**, например:
    - если коэффициенты ликвидности ниже порога — рекомендации по управлению оборотным капиталом;
    - если высокая долговая нагрузка — рекомендации по реструктуризации долга/ограничению заёмного финансирования;
    - если замедляется рост выручки — рекомендации по анализу продуктового портфеля/рынков.
  - [ ] Для каждой группы коэффициентов и типов выявленных рисков:
    - указать **тип рекомендаций** (что предложить сделать);
    - желаемую **структуру текста** (краткий заголовок + 1–2 фразы раскрытия).

- **A3.4. Шаблоны текстов рекомендаций**
  - [ ] Создать набор **шаблонов рекомендаций**, которые LLM сможет использовать и адаптировать:
    - с placeholders для чисел и категорий (например, `{ratio_name}`, `{ratio_value}`, `{risk_category}`);
    - с указанием желаемого тона (конструктивный, без паники, но честный по рискам).
  - [ ] Зафиксировать эти шаблоны в `docs/ANALYTICS_SPECS.md` в разделе «Рекомендации» или отдельном файле (по договорённости с капитаном).

#### 2.3. Программист 2 (бэкенд)

- **B3.1. Настройка взаимодействия с Ollama и выбор модели**
  - [ ] Установить и настроить **Ollama** на целевой машине разработки.
  - [ ] Выбрать и скачать базовую модель (например, `qwen2.5:7b-instruct` или `llama3.1:8b-instruct`), согласовав с командой.
  - [ ] Проверить **ручным запросом** (через `curl` или CLI), что модель:
    - корректно отвечает на русские и английские запросы;
    - может возвращать структурированный ответ (JSON‑подобный) по промпту.

- **B3.2. Реализация модуля `nlp.py`**
  - [ ] Реализовать функции (минимальный набор):
    - `analyze_explanatory_note(text: str) -> dict` или типизированную структуру, которая:
      - выделяет ключевые **риски** (список с категориями и краткими описаниями);
      - выделяет **перспективы/возможности**;
      - формирует **краткое резюме** пояснительной записки.
  - [ ] Реализовать **взаимодействие с Ollama**:
    - отправка промпта с инструкцией вернуть строго структурированный JSON;
    - разбор ответа и маппинг на внутренние структуры (списки рисков, перспектив, текст резюме).
  - [ ] Обработать типичные ошибки:
    - таймаут запроса;
    - неверный формат ответа (невалидный JSON);
    - отсутствие запущенного сервиса Ollama.
  - [ ] В случае ошибок предусмотреть **деградацию**:
    - логировать ошибку;
    - возвращать пустые списки/`None`, чтобы остальная часть пайплайна (числа и коэффициенты) продолжала работать.

- **B3.3. Реализация модуля `recommendations.py`**
  - [ ] Реализовать функцию (например) `generate_recommendations(ratios, score, nlp_insights) -> list[Recommendation]`, где:
    - вход:
      - рассчитанные коэффициенты и скор;
      - результаты `nlp.py` (риски/перспективы);
    - выход:
      - список структурированных рекомендаций: категория, приоритет, текст, связанные показатели/риски.
  - [ ] Реализовать **комбинированный подход**:
    - простые rule-based рекомендации на основе чисел (по правилам аналитика 2);
    - при необходимости — один вызов LLM для “полировки”/переформулировки рекомендаций в более читабельный текст.
  - [ ] Обеспечить, чтобы рекомендация всегда ссылалась хотя бы на:
    - конкретный коэффициент/группу коэффициентов и/или
    - конкретный риск/перспективу из NLP‑анализа.

- **B3.4. Реализация модуля `news.py` (опционально)**
  - [ ] При наличии времени и API‑ключа:
    - реализовать клиент к **NewsAPI** (или аналогичному сервису) для получения новостей по названию компании/тикеру;
    - ограничить количество новостей и период (например, 10–20 новостей за последние 30 дней).
  - [ ] Интегрировать библиотеку для оценки **сентимента** (например, Dostoevsky для русского языка):
    - для каждого заголовка/анонса определять тональность (позитив/негатив/нейтрал);
    - агрегировать общую картину настроений.
  - [ ] Определить формат данных, который будет возвращать `news.py` (список новостей с тэгом тональности).

- **B3.5. Обновление схем и эндпоинта `/analyze`**
  - [ ] В `backend/app/models/schemas.py`:
    - дополнить/уточнить поля, связанные с NLP и рекомендациями:
      - `nlp_summary: str | None`
      - `risks: list[...]`
      - `opportunities: list[...]`
      - `recommendations: list[...]`
      - (опц.) `news: list[...]`
  - [ ] В `backend/app/api/endpoints.py`:
    - расширить пайплайн:
      - после расчёта коэффициентов и скора вызывать `nlp.py` и `recommendations.py`;
      - при наличии модуля новостей — вызывать и его (с возможностью отключения через конфиг/флаг).
    - гарантировать, что при ошибке LLM/новостей числовая часть ответа остаётся работоспособной.

- **B3.6. Docker‑подготовка**
  - [ ] Создать `infra/docker/backend.Dockerfile` с:
    - установкой Python‑зависимостей;
    - установкой Tesseract и необходимых системных библиотек для pdfplumber;
    - настройкой переменных окружения.
  - [ ] Создать базовый `infra/docker/docker-compose.yml` со службами:
    - `backend` (FastAPI‑сервис);
    - (опц.) `frontend`, если его логично поднимать из Docker;
    - `ollama` или предположить внешний запуск Ollama (описать это в комментариях).
  - [ ] Проверить, что:
    - `docker-compose up backend` поднимает бэкенд, доступный с хоста;
    - взаимодействие бэкенда с Ollama работает в контейнеризированной конфигурации (или задокументировать особенности).
