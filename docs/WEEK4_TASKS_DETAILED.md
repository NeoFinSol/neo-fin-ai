## Неделя 4 — тестирование, полировка, документация и подготовка к защите

Этот документ конкретизирует задачи недели 4 и фокусируется на:
- модульном и интеграционном тестировании;
- полировке UX/UI и стабильности пайплайна;
- финальной документации и запуске через Docker;
- подготовке презентации и сценария демо.

### 1. Цели недели 4 (уточнение)

- **Стабильность**: система должна предсказуемо работать на выбранных демо‑отчётах и не “падать” на типичных ошибках ввода.
- **Качество**: основные модули покрыты хотя бы базовыми тестами, ключевые баги устранены.
- **Презентабельность**: UI аккуратный, понятный, без “сырого” вида, понятный пользовательский сценарий.
- **Готовность к защите**: есть презентация, сценарий демо, отрепетированный показ.

### 2. Задачи по ролям (детализация)

#### 2.1. Аналитик 1 (данные)

- **A4.1. Финальное тестирование качества извлечения и коэффициентов**
  - [ ] Прогнать через систему все доступные отчёты из `data/pdf_samples/`.
  - [ ] Для каждого отчёта:
    - сравнить извлечённые показатели с ручной разметкой (см. материалы недели 2);
    - оценить корректность рассчитанных коэффициентов и скора.
  - [ ] Зафиксировать:
    - типичные ошибки (например, неверно определённые единицы измерения, пропущенные строки, перепутанные знаки);
    - сложные кейсы (нестандартные таблицы, смена формата).
  - [ ] Сформировать короткий отчёт по качеству (таблица/документ в `docs/`), где:
    - указать долю корректных извлечений по каждому ключевому показателю;
    - описать наиболее проблемные места.

#### 2.2. Аналитик 2 (рынок, бизнес-модель)

- **A4.2. Калибровка скоринга и рекомендаций**
  - [ ] На основе результатов тестирования:
    - для нескольких компаний оценить адекватность скоринга (интуитивно “сильная/слабая” компания имеет более высокий/низкий скор);
    - проверить, насколько рекомендации соответствуют реальной картине (нет ли очевидно нелепых советов).
  - [ ] При необходимости:
    - скорректировать весовые коэффициенты и пороговые значения для отдельных коэффициентов;
    - доработать шаблоны рекомендаций (убрать повторяющиеся/слишком общие формулировки).
  - [ ] Обновить соответствующие разделы в `docs/ANALYTICS_SPECS.md`:
    - финальные веса для скоринга;
    - актуальные шаблоны/примеры рекомендаций.

#### 2.3. Программист 2 (бэкенд)

- **B4.1. Модульные тесты для основных модулей**
  - [ ] Добавить (или дописать) юнит‑тесты как минимум для:
    - `ocr.py`:
      - тесты на обработку текстовых и сканированных PDF (по возможности через заглушки/моки, чтобы не зависеть от реального OCR в каждом тесте);
      - обработку ошибок (недоступность Tesseract, некорректный файл).
    - `extractor.py`:
      - тесты на корректное извлечение отдельных показателей на основе заранее подготовленных текстовых фрагментов;
      - кейсы с разными форматами чисел (тыс./млн, запятая/точка, пробелы).
    - `ratios.py`:
      - тесты на правильность расчёта коэффициентов по synthetic‑наборам показателей;
      - проверка отсутствия падений при делении на ноль/отсутствующих данных.
    - `scoring.py`:
      - тесты, где заранее задан набор коэффициентов и ожидаемый скор (несколько сценариев: “хорошая”, “средняя”, “слабая” компания).
    - `recommendations.py`, `nlp.py`:
      - тесты с фейковыми/моковыми данными (без реальных вызовов LLM и внешних API).

- **B4.2. Интеграционные тесты и отладка пайплайна**
  - [ ] Реализовать (или описать и частично реализовать) интеграционные тесты:
    - сквозной вызов `/analyze` на нескольких PDF и проверка ключевых полей ответа (наличие показателей, коэффициентов, скора, NLP‑результатов).
  - [ ] Починить/обойти критичные баги, выявленные аналитиками и в ходе тестирования.
  - [ ] Улучшить обработку ошибок:
    - более информативные сообщения в логах;
    - аккуратные ответы клиенту (без “traceback” наружу).

- **B4.3. Рефакторинг и финальная уборка**
  - [ ] Навести порядок в структуре модулей:
    - убрать неиспользуемый код;
    - при необходимости вынести вспомогательные функции в отдельные утилити‑модули.
  - [ ] Проверить стиль кода и базовое соответствие принятым линтерам/форматированию (если они есть).

- **B4.4. Документация и запуск**
  - [ ] Обновить `README.md`:
    - кратко описать функциональность проекта;
    - добавить инструкцию по запуску в режиме разработки (локально) и через Docker/`docker-compose`;
    - привести примеры запросов/ответов (JSON) для основного эндпоинта `/analyze`.
  - [ ] Добавить `.env.example` (если его ещё нет или он не актуален) с перечнем необходимых переменных окружения (ключи, URL, настройки LLM).
  - [ ] Проверить и при необходимости обновить `infra/docker/*`:
    - чтобы `docker-compose up` поднимал работоспособный стек (как минимум бэкенд + зависимости).

#### 2.4. Программист 1 (фронтенд)

- **F4.1. Полировка UI/UX**
  - [ ] Пройтись по основным сценариям использования и:
    - убедиться, что форма загрузки PDF удобна и понятна;
    - результаты анализа (показатели, коэффициенты, скор, риски, рекомендации, новости) аккуратно и логично расположены.
  - [ ] Улучшить визуальную часть:
    - выравнивание таблиц, отступы, заголовки секций;
    - использование простых, но приятных цветовых схем;
    - (опц.) добавить графики по динамике ключевых показателей/коэффициентов (например, через Chart.js или простые SVG/Canvas‑диаграммы).

- **F4.2. Обработка ошибок и “краевых” сценариев**
  - [ ] Проверить, как фронтенд реагирует на:
    - загрузку нескольких разных файлов подряд;
    - ошибки со стороны бэкенда (некорректный PDF, проблемы LLM, отсутствие сети).
  - [ ] Добавить:
    - понятные сообщения об ошибках;
    - подсказки пользователю (какие форматы отчётов лучше поддерживаются, пример “удачного” отчёта).

### 3. Интеграционное тестирование и чек-листы

- **Интеграционные тесты**:
  - [ ] Подготовить несколько “демо‑наборов”:
    - один отчёт с максимально “красивым” сценарием (всё извлекается корректно);
    - один/два более сложных отчёта (скан, странный формат таблиц и т.п.).
  - [ ] Для каждого демо‑набора:
    - прописать ожидаемое поведение системы (какие ключевые числа, примерный скор, характер рекомендаций);
    - прогнать через систему и зафиксировать результат (для дальнейших ссылок на защите).

- **Чек-лист качества (обновлённый к финалу)**:
  - [ ] Загрузка PDF работает стабильна (3+ разных отчёта подряд, без перезапуска).
  - [ ] Ключевые показатели и коэффициенты для демо‑отчётов выглядят адекватно.
  - [ ] Скор по 100‑балльной шкале различает “сильные” и “слабые” кейсы.
  - [ ] NLP‑анализ (если включён) даёт осмысленные риски/перспективы и сжатое резюме.
  - [ ] Рекомендации ссылаются на конкретные показатели и риски.
  - [ ] Docker‑сборка собирается и поднимается без ручного шаманства.

### 4. Подготовка к защите

- **Презентация (10–15 слайдов)**:
  - [ ] Подготовить слайды по структуре из общего плана (проблема → решение → архитектура → функциональность → качество и ограничения → планы развития → команда).
  - [ ] Вставить в презентацию:
    - скриншоты интерфейса;
    - схему архитектуры модулей;
    - пару примеров “до/после” (ручной анализ vs ассистент).

- **Демо‑сценарий**:
  - [ ] Прописать сценарий живого демо:
    - какой отчёт показывается;
    - какие действия выполняет демонстратор (куда нажимает, что комментирует);
    - где делается акцент (скоринг, рекомендации, NLP‑анализ и т.д.).
  - [ ] Распределить роли в команде:
    - кто рассказывает про бизнес‑часть;
    - кто про архитектуру и технологии;
    - кто комментирует демо и отвечает на технические вопросы.

- **Репетиции и финальный чек**:
  - [ ] Провести минимум 2–3 прогонки демо:
    - один раз “для себя”;
    - один раз максимально близко к реальному таймингу защиты.
  - [ ] Убедиться, что:
    - все нужные сервисы (включая LLM/Ollama) запускаются заранее и не требуют долгой прогрузки во время живого выступления;
    - есть “план Б” на случай проблем с интернетом или внешними API (например, офлайн‑демо на заранее прогнанных JSON‑результатах).
