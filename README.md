# НеоФин.Документы

## Краткое определение проекта

`НеоФин.Документы` — система, которая превращает неструктурированную финансовую отчётность в проверяемый аналитический результат.  
Это не просто загрузка PDF и не «калькулятор коэффициентов»: платформа извлекает данные из сложных документов, фиксирует происхождение чисел, выполняет детерминированный расчёт и формирует сопоставимый профиль компании для обоснованных решений.

Проект развивается как продуктовый модуль экосистемы `НеоФин.Контур` и ориентирован на качество анализа, прослеживаемость и воспроизводимость результата.

## Для чего нужен проект

Финансовая отчётность в PDF часто неудобна для практической аналитики: данные разнесены по таблицам, формат нестабилен, часть документов приходит в виде сканов.  
`НеоФин.Документы` закрывает этот разрыв между документом и анализом: переводит исходный файл в структурированный набор показателей, пригодный для сравнения периодов и последующей экспертизы.

Система:

- снижает объем ручного переноса чисел из PDF;
- сокращает время первичного разбора отчётов;
- повышает сопоставимость результатов между периодами;
- сохраняет прослеживаемость происхождения данных в итоговом результате.

## Какие задачи решает система

- приём и обработка бухгалтерской и финансовой отчётности в PDF;
- извлечение числовых показателей из текста, таблиц и сканов;
- расчёт коэффициентов и интегральной оценки риска;
- многопериодный анализ в одной сессии;
- выдача машинно-обрабатываемого результата через прикладной интерфейс;
- выдача пользовательского отчёта в веб-интерфейсе;
- формирование текстовых рекомендаций при доступности настроенного поставщика языковой модели.

## Ключевые возможности

- асинхронная обработка задач с режимами `background` и `celery`;
- обновление статусов в реальном времени через WebSocket;
- контроль качества извлечения на уровне каждого показателя (`confidence`, `source`, `reason_code`);
- расчёт интегрального показателя `0-100` с прозрачной расшифровкой факторов;
- поддержка профилей скоринга (`auto`, `generic`, `retail_demo`);
- диагностика состояния системы через служебные маршруты `/system/*`;
- отказоустойчивая работа языкового слоя с повторными попытками, тайм-аутами и механизмом защиты от каскадных сбоев.

Практический результат — переход от исходного документа к объяснимому расчётному профилю, который можно проверять, сравнивать и использовать в рабочем контуре.

## Как проходит анализ документа

1. Пользователь загружает PDF в маршрут `/upload`.
2. Сервер ставит задачу в выполнение и возвращает `task_id`.
3. Подсистема извлечения читает документ, извлекает показатели и метаданные достоверности.
4. Математический слой рассчитывает производные метрики, коэффициенты и интегральный скоринг.
5. При доступности языкового поставщика формируется текстовый аналитический блок.
6. Результат сохраняется в базе данных и возвращается по `/result/{task_id}`.
7. Интерфейс получает статусы через `/ws/{task_id}` и/или периодический опрос.

## Основные подсистемы

### Серверная часть

- язык: Python (`python:3.11-slim` в сборке);
- веб-каркас: FastAPI;
- хранение: PostgreSQL + SQLAlchemy + Alembic;
- очереди и длительные задачи: встроенный фоновый режим или Celery + Redis;
- ключевые маршруты: загрузка/результат, история анализов, многопериодный анализ, системное состояние, WebSocket-канал.

### Пользовательская часть

- стек: React 19 + TypeScript + Mantine + Vite;
- основной сценарий: загрузка отчётности, отслеживание статусов, просмотр детального отчёта;
- дополнительные сценарии: история анализов; сравнение периодов в интерфейсе при многопериодной сессии (см. `frontend/README.md`); выбор поставщика языковой модели.

### Подсистема извлечения данных

- извлечение из текста и таблиц PDF;
- OCR-контур для сканированных документов;
- присвоение достоверности каждому извлеченному значению;
- формирование трассировки решений для объяснимого разбора результата.

Подсистема ориентирована на работу не только с «идеальными» файлами, но и с неоднородной отчётностью, где качество входного документа заранее не гарантировано.

### Математический слой

В проекте используется математический слой второго поколения (Math Layer v2) как действующая основа расчётов и утвержденное направление развития:

- реестр метрик и типизированные входы;
- формализованные состояния валидности и коды причин;
- единые правила расчёта, нормализации и интерпретации;
- согласование расчётного контура со скорингом и объяснимостью.

Этапные инженерные документы разработки (wave/freeze) не используются как основа публичного описания продукта.

## Архитектура на верхнем уровне

Система разделена на четыре функциональных уровня:

1. **Транспортный уровень** — прием файлов, выдача статусов, прикладные маршруты, WebSocket.
2. **Уровень обработки** — извлечение данных, проверка качества, преобразование в внутренние структуры.
3. **Математический уровень** — расчёт метрик, коэффициентов, интегральной оценки и факторов влияния.
4. **Уровень представления** — сохранение результата, выдача в прикладной интерфейс и отображение в веб-интерфейсе.

Языковой аналитический модуль дополняет числовой результат, но не заменяет детерминированный расчётный контур.

## Что отличает проект

- объяснимый результат: система возвращает не только итоговые показатели, но и происхождение исходных чисел;
- детерминированный расчёт как основа: числовой результат не зависит от доступности языковой модели;
- разделение контуров извлечения, расчёта и языкового анализа, что упрощает проверку и развитие системы;
- зрелый математический слой второго поколения, который задает единые правила интерпретации и скоринга;
- устойчивый контур длительных задач с отменой, восстановлением и наблюдаемостью;
- единый путь от неструктурированного PDF к сопоставимому аналитическому профилю компании.

## Текущее состояние проекта

Проект находится в стадии активного продуктового развития:

- базовый контур анализа, скоринга и многопериодной обработки реализован;
- интерфейсы прикладного доступа и пользовательского просмотра работают в едином контуре;
- математический слой второго поколения принят как основа развития расчётной части;
- часть направлений продолжает развиваться и фиксируется в профильных технических документах.

Исторический конкурсный контур сохранен как архив и не задает основную продуктовую логику документа.

## Запуск / локальная разработка

Минимальный локальный запуск проверяется через Docker Compose.

1. Подготовить `.env` на основе `.env.example`.
2. Запустить стек:

```bash
docker compose up --build
```

3. После запуска:

- веб-интерфейс: `http://localhost`
- сервер: `http://localhost:8000`

Стек по умолчанию в `docker-compose.yml` включает **PostgreSQL**, **Redis**, **worker Celery** и фронтенд за nginx — это не «один процесс в foreground», а контур, близкий к полноценной локальной эксплуатации (подробнее — в `docs/INSTALL_WINDOWS.md` и `docs/CONFIGURATION.md`).

Дополнительные профили:

- `test` — отдельная тестовая база данных;
- `ollama` — локальный сервис языковой модели.

Подробная настройка параметров среды и режимов запуска описана в `docs/CONFIGURATION.md` и `docs/INSTALL_WINDOWS.md`.
